## Web Hosting 

<b>웹 호스팅(Web hosting)</b>은 웹사이트를 인터넷에서 볼 수 있도록 하는 서비스이다. 웹 호스팅 회사는 고객이 제작한 웹사이트를 자신의 서버에 저장하고, 인터넷을 통해 전 세계에 공개한다.

일반적으로, 웹 호스팅을 사용하는 경우, 고객은 자신의 도메인 이름을 등록하고 호스팅 서비스에서 제공하는 서버에 웹사이트 파일을 업로드한다. 호스팅 서비스는 이 파일을 공개할 수 있도록 인터넷에 연결된 서버에서 처리하며, 이를 통해 인터넷 사용자들이 고객의 웹사이트를 방문하고 정보를 볼 수 있다.

웹 호스팅은 기업, 개인, 단체 등 다양한 사용자가 사용할 수 있으며, 일반적으로 월별 또는 연간 요금을 지불하여 사용한다. 호스팅 서비스는 보안, 백업, 데이터 복구 등 다양한 기능을 제공하며, 고객은 필요에 따라 호스팅 플랜을 선택할 수 있다.


## Internet Protocols


<b>인터넷 프로토콜(Internet Protocols)</b>은 인터넷에서 데이터 통신을 위해 사용되는 프로토콜의 모음. 

* <b>TCP (Transmission Control Protocol)</b> </br>
: 데이터의 안정적인 전송을 보장하는 연결형 프로토콜

* <b>IP (Internet Protocol)</b> </br>
: 인터넷상에서 패킷을 전송하는 데 사용되는 프로토콜. IP는 패킷을 보내는 데 사용되는 주소 체계를 정의한다.

* <b>HTTP (Hypertext Transfer Protocol)</b> </br>
: 웹 페이지에서 데이터를 전송하는 데 사용되는 프로토콜. 웹 브라우저와 서버 간의 통신에 사용된다.

* <b>FTP (File Transfer Protocol)</b> </br>
: 파일 전송을 위해 사용되는 프로토콜. 원격 호스트 간에 파일을 전송할 수 있다.

* <b>SMTP (Simple Mail Transfer Protocol)</b> </br>
: 이메일을 전송하는 데 사용되는 프로토콜. 이메일 클라이언트와 이메일 서버 간의 통신에 사용한다.

* <b>POP (Post Office Protocol)</b> </br>
: 이메일 클라이언트에서 이메일 서버로부터 이메일을 가져오는 데 사용되는 프로토콜.

* <b>IMAP (Internet Message Access Protocol)</b> </br>
: 이메일 서버에서 이메일을 읽고 관리하는 데 사용되는 프로토콜.

* <b>DNS (Domain Name System)</b> </br>
: 도메인 이름과 IP 주소 간의 매핑을 제공하는 프로토콜.


## HTTP
HTTP 메소드는 서버에 요청을 보내는 방법을 지정하는데 사용된다.
<h3>HTTP 메소드</h3>

* <b>GET</b> </br>
: 서버로부터 데이터를 검색하기 위해 사용된다. 예를 들어, 웹 페이지나 이미지를 다운로드할 때 사용.

* <b>POST</b> </br>
: 서버에 데이터를 제출하기 위해 사용된다. 예를 들어, 로그인 또는 회원가입 양식을 제출할 때 사용.

* <b>PUT</b> </br>
: 서버에 데이터를 업데이트하기 위해 사용된다. 예를 들어, 이미지나 파일을 업로드하거나, 기존 데이터를 업데이트할 때 사용.

* <b>DELETE</b> </br>
: 서버에서 데이터를 삭제하기 위해 사용된다. 예를 들어, 데이터베이스에서 레코드를 삭제할 때 사용.

<h3>HTTP 상태 코드</h3>

* <b>1XX (Informational)</b> </br>
: 클라이언트 요청이 수신되었으며 처리 중임을 나타낸다.
 </br>100 - Continue (클라이언트 요청의 일부를 서버가 받았으며, 계속해서 요청을 보내도 됨을 나타냄),
 </br>101 - Switching Protocols (서버가 클라이언트의 upgrade 요청을 수락하고, 프로토콜을 바꿀 것임을 나타냄)

* <b>2XX (Successful)</b> </br>
: 클라이언트 요청이 성공적으로 처리되었음을 나타낸다.
 </br>200 - OK (요청이 성공적으로 처리되었으며, 응답으로 요청한 자원을 전달함),
 </br>201 - Created (요청이 성공적으로 처리되었으며, 새로운 자원이 생성되었음을 나타냄),
 </br>204 - No Content (요청이 성공적으로 처리되었지만, 서버가 별도의 응답 본문을 제공하지 않음)

 
* <b>3XX (Redirection)</b> </br>
: 클라이언트가 추가 조치를 취해야 함을 나타낸다.
 </br>301 - Moved Permanently (요청한 자원이 다른 URL로 옮겨졌음을 나타냄),
 </br>302 - Found (요청한 자원이 다른 URL에 일시적으로 이동되었음을 나타냄),
 </br>304 - Not Modified (클라이언트가 이미 최신 버전의 자원을 가지고 있음을 나타내고, 서버가 자원이 수정되지 않았음을 알리는 응답)
 
 
* <b>4XX (Client Error)</b> </br>
: 클라이언트 요청이 잘못되었거나, 요청한 자원이 없음을 나타낸다.
 </br>400 - Bad Request (요청이 잘못되었거나, 서버에서 처리할 수 없음을 나타냄),
 </br>403 - Forbidden (서버가 요청을 거부함을 나타냄),
 </br>404 - Not Found (요청한 자원이 서버에 없음을 나타냄)
 
 * <b>5XX (Server Error)</b> </br>
: 서버에서 처리 중 오류가 발생했음을 나타낸다.
 </br>500 - Internal Server Error (서버에서 처리 중 오류가 발생함을 나타냄),
 </br>503 - Service Unavailable (tj버가 일시적으로 요청을 처리할 수 없음)

## Libraries & Frameworks 
두 개념 모두 일반적인 문제들을 더 쉽게 해결하기 위해 다른 개발자들이 작성한 재사용 가능한 코드다.

<h3>Libraries</h3>
코드를 재사용하기 위해 다른 프로그램에서 호출할 수 있는 함수, 클래스, 메소드 등의 집합이다.</br>
라이브러리를 사용할 때는 어플리케이션 흐름을 직접 제어해야 한다.

<h3>Frameworks</h3>
프레임워크는 전체적인 구조와 흐름을 정해놓은 틀이라고 할 수 있다.</br>
프레임워크는 개발자가 작성한 코드를 프레임워크에서 호출하여 실행된다.</br></br>

> 라이브러리는 개발자가 필요할 때 호출하여 사용하는 도구이고,    
> 프레임워크는 개발자가 작성하는 코드를 호출하여 실행하는 도구이며 코드를 작성하는 방법에 대한 구조를 제공한다.


## API & JSON

<h3>API (Application Programming Interface)</h3>
응용 프로그램 간 상호 작용을 위한 인터페이스를 말합니다
API는 서로 다른 두 소프트웨어 시스템 사이에서 데이터를 전송하고 서비스를 공유하기 위한 규칙, 프로토콜, 도구 및 문서를 제공한다.
이를 통해 개발자들은 서로 다른 소프트웨어 시스템에서 작성된 기능을 호출하고, 데이터를 검색하고, 다양한 서비스를 활용할 수 있다.
</br>
<h4> REST(Representational State Transfer) API </h4>

> - 웹 어플리케이션에서 클라이언트와 서버 간의 통신을 위한 아키텍처 스타일이다.</br>
> - 웹 / 앱 페이지 안에서 하는 행동을 크게 4가지로 나누어 약속함
> > 1. GET 메소드: 리소스를 조회할 때 사용
> > 2. POST 메소드: 새로운 리소스를 생성할 때 사용
> > 3. PUT 메소드: 기존 리소스를 업데이트할 때 사용
> > 4. DELETE 메소드: 리소스를 삭제할 때 
> - 주된 특징은 무상태(서버가 요청 간에 클라이언트 데이터를 저장하지 않음)이다. 

<h3>JSON (JavaScript Object Notation)</h3>
데이터를 표현하는 포맷 중 하나</br>
이름-값 쌍으로 이루어진 데이터 객체를 사용하여 데이터 표현

```javascript
{
  "name": "John",
  "age": 30,
  "hobbies": ["reading", "swimming", "traveling"],
  "address": {
    "street": "123 Main St",
    "city": "New York",
    "state": "NY"
  }
}
```

## Cookie & Session & Token & JWT

<h3>Cookie</h3>
쿠키는 서버가 클라이언트에게 보내는 작은 데이터 조각이다. </br>
주로 브라우저에 저장되며, 웹사이트 방문 시 사용자 인증 정보, 선호하는 언어, 장바구니 등을 저장한다.</br>
보안적인 취약점이 있을 수 있으므로, 쿠키에 중요한 정보를 저장하거나, 암호화되지 않은 평문으로 저장하는 것은 권장하지 않는다.

<h3>Session</h3>
세션은 서버에 저장되는 사용자 정보의 집합이다. </br>
웹 서버는 세션 ID를 발급하고, 이를 쿠키나 URL 매개변수에 담아 클라이언트에게 보낸다. </br>
클라이언트가 이후 요청을 보낼 때마다 세션 ID를 함께 보내면, 서버는 이를 확인하여 사용자 정보를 유지한다.</br>
세션은 쿠키보다 더 안전하며, 중요한 정보를 저장할 수 있다.

<h3>Token</h3>
토큰은 인증 정보를 안전하게 전송하기 위한 방법 중 하나이다.</br>
일반적으로 JSON 형태로 구성되며, 서버는 토큰을 생성하고, 클라이언트에게 전달한다.</br>
클라이언트는 이후 요청에서 토큰을 함께 보내면, 서버는 이를 검증하여 사용자를 인증한다. </br>
토큰은 보안적인 이유로 SSL/TLS 등의 암호화된 연결을 통해 전송되어야 한다.

<h3>JWT</h3>
JWT는 토큰 중 하나로, JSON 형태로 인코딩된 정보를 안전하게 전달하기 위해 사용된다.</br>
JWT는 서버와 클라이언트 간에 인증을 수행하는 데 사용된다.</br>
JWT는 일반적으로 세션과 같은 방식으로 작동하지만, 세션과 달리 서버에 저장할 필요가 없으므로 서버의 확장성과 유연성이 향상된다. </br></br>

> 쿠키는 클라이언트 측에 저장되며, </br>
> 세션은 서버 측에 저장되며, </br>
> 토큰 및 JWT는 인증 정보를 안전하게 전송하기 위한 방법
